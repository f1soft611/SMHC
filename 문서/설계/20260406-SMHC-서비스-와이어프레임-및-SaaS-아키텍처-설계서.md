# SMHC 서비스 와이어프레임 및 SaaS 아키텍처 설계서

작성일: 2026-04-06  
대상: 대표 및 의사결정자 회의용  
범위: 현재 frontend 구현 기준 화면 구조, 사용자 흐름, SaaS/클라우드 설계 구조 정리

## 1. 문서 목적

본 문서는 Smart HACCP Cloud 프로젝트의 현재 서비스 구조를 대표 관점에서 빠르게 이해할 수 있도록 정리한 설계 자료입니다.

이번 회의에서 확인해야 할 핵심은 아래 3가지입니다.

1. 현재 어떤 화면이 준비되어 있고 어떤 업무 흐름을 지원하는가
2. 이 프로젝트가 SaaS형 클라우드 서비스로 확장 가능한 구조인가
3. 운영 관점에서 권한, 테넌트, 데이터 흐름이 어떻게 분리되어 있는가

## 2. 서비스 한 줄 정의

SMHC는 식품 제조 현장의 HACCP 점검, CCP 모니터링, 운영관리, 보고 체계를 클라우드 기반으로 통합하는 Smart HACCP SaaS 플랫폼입니다.

현재 구현은 다음 방향에 초점을 맞추고 있습니다.

- 현장 점검과 센서 기반 모니터링을 한 화면 흐름으로 연결
- 사용자와 권한을 역할 기반으로 통제
- 사업장과 고객사를 구분할 수 있는 테넌트 구조 반영
- 기능 도메인별 독립 확장이 가능한 프론트 구조 확보

## 3. 현재 구현 범위 요약

### 3.1 현재 사용 가능한 화면

| 구분 | 화면                   | 목적                                           | 상태      |
| ---- | ---------------------- | ---------------------------------------------- | --------- |
| 인증 | 로그인                 | 사용자 인증 및 테넌트 진입                     | 구현 완료 |
| 메인 | CCP 통합 관제 대시보드 | 점검 상태, 지연 항목, 동기화율, 공정 현황 조회 | 구현 완료 |
| 운영 | 사용자관리             | 사용자 검색, 필터, 등록, 수정, 삭제            | 구현 완료 |
| 예외 | 접근 권한 없음         | 권한 부족 시 차단 화면                         | 구현 완료 |
| 확장 | 준비중 화면            | 향후 메뉴 기능의 placeholder                   | 구현 완료 |

### 3.2 향후 확장 예정 영역

- 실시간 CCP 모니터링 현황
- 가열공정 데이터 수집 및 개선일지
- 냉동/냉각 공정 데이터 수집 및 개선일지
- 선행요건 점검 일지
- HACCP 조직도 및 교육 훈련 대장
- 생산 및 품질 종합 대시보드
- CCP 트렌드 분석 및 이상징후 리포트
- 권한관리

즉, 현재 구조는 핵심 뼈대가 먼저 완성되어 있고, 개별 업무 모듈을 점진적으로 확장하는 방식에 적합하게 설계되어 있습니다.

## 4. 전체 서비스 구조

### 4.1 상위 서비스 맵

```text
[사용자 인증]
    -> [메인 대시보드]
    -> [업무 메뉴 선택]
    -> [운영 화면 또는 준비중 화면]
    -> [권한 검증 및 데이터 조회]
```

### 4.2 메뉴 기반 정보 구조

현재 서비스 메뉴는 아래 5개 축으로 설계되어 있습니다.

1. Smart HACCP 모니터링
2. CCP 관리
3. HACCP 문서 관리
4. 데이터 분석 및 보고
5. 시스템 설정

이 구조는 단순한 메뉴 구성이 아니라 서비스 사업 구조를 그대로 반영한 정보 구조입니다.

- 모니터링: 실시간 운영 가시화
- 관리: 현장 실행과 조치 기록
- 문서: 감사 대응 및 이력 관리
- 분석: 경향 파악과 리포트화
- 설정: 사용자 및 권한 운영

대표 관점에서 보면, 현재 프로젝트는 단일 화면 개발이 아니라 운영 SaaS 플랫폼의 기본 골격을 먼저 세운 상태입니다.

## 5. 와이어프레임 구성

### 5.1 공통 레이아웃

모든 핵심 화면은 동일한 운영 레이아웃을 공유합니다.

```text
┌──────────────────────────────────────────────┐
│ HeaderBar                                    │
│ - 메뉴 토글                                  │
│ - 서비스 로고                                │
│ - 검색 영역                                  │
│ - 사용자 정보                                │
│ - 로그아웃                                   │
├──────────────────────────────────────────────┤
│ MenuDrawer                                   │
│ - 역할 기반 카테고리 메뉴                    │
│ - 현재 사용 가능 / 준비중 상태 표시          │
├──────────────────────────────────────────────┤
│ Page Content                                 │
│ - 화면별 핵심 업무 영역                      │
└──────────────────────────────────────────────┘
```

이 공통 레이아웃의 의미는 다음과 같습니다.

- 사용자는 항상 같은 구조 안에서 업무를 이동한다
- 권한에 따라 노출되는 메뉴가 달라진다
- 기능이 늘어나도 전체 UX 구조는 흔들리지 않는다

### 5.2 로그인 화면 와이어프레임

목적: 사용자 인증 후 해당 테넌트의 운영 환경으로 진입

```text
┌──────────────────────────────────────┐
│ 로그인 배경 영역                     │
│                                      │
│   ┌──────────────────────────────┐   │
│   │ SMHC Access                  │   │
│   │ 설명 문구                    │   │
│   │ Tenant / ID / Password 입력  │   │
│   │ 로그인 버튼                  │   │
│   └──────────────────────────────┘   │
└──────────────────────────────────────┘
```

핵심 포인트:

- 단순 로그인 화면이 아니라 tenantCode 기반 진입 구조를 전제로 함
- 고객사 또는 사업장 단위 서비스 구분의 시작점 역할 수행

### 5.3 메인 대시보드 와이어프레임

목적: 현장 운영자가 가장 먼저 확인해야 하는 위험 신호와 상태를 한 화면에 요약

```text
┌──────────────────────────────────────────────────────────────┐
│ 상단 정보 영역                                               │
│ - 사이트 선택                                                │
│ - 날짜 범위 선택                                             │
│ - 업체 / 사용자 / 사업장 정보                                │
├──────────────────────────────────────────────────────────────┤
│ KPI 카드 영역                                                │
│ - 검증 완료 점검                                             │
│ - 지연 점검                                                  │
│ - 클라우드 동기화율                                          │
├──────────────────────────────────────────────────────────────┤
│ 우선 처리 작업 영역                                          │
│ - 지연 점검 후속 처리                                        │
│ - 센서 데이터 관찰                                           │
│ - 공정별 현장 이슈                                           │
├──────────────────────────────────────────────────────────────┤
│ 공정별 리포트 영역                                           │
│ - 공정 탭                                                    │
│ - 준수율                                                     │
│ - 오픈 이슈 수                                               │
├──────────────────────────────────────────────────────────────┤
│ 차트 영역                                                    │
│ - 공정별 준수율 추이                                         │
│ - 동기화율 추이                                              │
└──────────────────────────────────────────────────────────────┘
```

핵심 포인트:

- 운영자가 오늘 처리해야 할 항목을 우선순위 중심으로 파악 가능
- 사업장 선택과 날짜 범위 선택을 통해 멀티 사이트 운영에 대응
- 센서 데이터, 점검 데이터, 조치 흐름을 하나의 운영 화면에서 연결

### 5.4 사용자관리 화면 와이어프레임

목적: 관리자 권한 사용자가 사용자 계정과 운영 권한을 관리

```text
┌──────────────────────────────────────────────────────────────┐
│ 페이지 헤더                                                  │
│ - 사용자관리 제목                                            │
│ - 새 사용자 추가 버튼                                        │
├──────────────────────────────────────────────────────────────┤
│ 검색 및 필터 영역                                            │
│ - 이름/이메일 검색                                           │
│ - 부서 필터                                                  │
│ - 역할 필터                                                  │
│ - 상태 필터                                                  │
├──────────────────────────────────────────────────────────────┤
│ 사용자 목록 테이블                                           │
│ - 정렬                                                       │
│ - 페이지네이션                                               │
│ - 수정 / 삭제 액션                                           │
├──────────────────────────────────────────────────────────────┤
│ 사용자 등록/수정 다이얼로그                                  │
│ - 이름, 이메일, 부서, 역할, 상태                             │
└──────────────────────────────────────────────────────────────┘
```

핵심 포인트:

- 시스템 설정은 일반 사용자 영역과 분리된 관리자 전용 운영 공간
- 향후 권한관리, 기준값 관리, 승인 체계 관리까지 확장 가능한 구조

### 5.5 예외 및 준비중 화면

#### 접근 권한 없음

- 인증은 되었지만 권한이 없는 경우 차단
- 운영 보안 구조가 화면 레벨에서도 명확히 작동함을 보여줌

#### 준비중 화면

- 미구현 기능을 막연히 숨기지 않고 서비스 로드맵으로 보여줌
- 대표 회의에서는 확장 예정 모듈 설명에 활용 가능

## 6. 사용자 흐름 설계

### 6.1 기본 사용자 흐름

```text
로그인
  -> 인증 성공
  -> 대시보드 진입
  -> 메뉴 선택
  -> 권한 확인
  -> 화면 진입 또는 접근 제한
```

### 6.2 관리자 흐름

```text
관리자 로그인
  -> 대시보드 진입
  -> 시스템 설정 진입
  -> 사용자 검색/등록/수정/삭제
  -> 권한 운영 관리
```

### 6.3 대표 관점 해석

이 흐름은 다음 3개 레벨의 운영 통제를 동시에 보여줍니다.

1. 인증 통제: 로그인 사용자만 진입 가능
2. 권한 통제: 역할별로 메뉴와 화면 접근 범위 제한
3. 운영 통제: 사용자는 자신의 업무 범위 안에서만 기능 수행

## 7. SaaS 및 클라우드 아키텍처 설계

### 7.1 설계 방향

현재 프론트 구조는 단일 프로젝트형 화면 묶음이 아니라 SaaS형 업무 플랫폼으로 성장하기 위한 구조를 이미 반영하고 있습니다.

설계 방향은 아래와 같습니다.

- 공통 인프라와 개별 업무 기능을 분리
- 서버 상태와 UI 상태를 분리
- 테넌트와 권한을 화면 진입 단계부터 반영
- API 호출을 하나의 표준 클라이언트로 통합
- 기능 도메인별 독립 확장이 가능하도록 feature 구조 채택

### 7.2 5계층 구조

```text
[1] 사용자 인터페이스 계층
    페이지, 카드, 테이블, 폼, 차트

[2] 라우팅 및 접근 통제 계층
    로그인 여부 확인, 역할 검증, 경로 제어

[3] 도메인 기능 계층
    auth, monitoring, navigation, system-settings

[4] 상태 및 데이터 계층
    TanStack Query, Zustand

[5] API 연결 계층
    apiClient, 토큰 주입, 백엔드 API 호출
```

### 7.3 계층별 설명

#### 1) 사용자 인터페이스 계층

- 페이지와 컴포넌트가 실제 사용자 경험을 담당
- 로그인, 대시보드, 사용자관리, 예외 화면으로 구성
- 공통 디자인 시스템을 사용해 브랜드 일관성 유지

#### 2) 라우팅 및 접근 통제 계층

- 인증 여부에 따라 로그인 또는 메인으로 분기
- 권한이 없는 사용자는 접근 제한 화면으로 이동
- 메뉴 경로와 실제 라우트를 동기화하여 운영 혼선을 줄임

#### 3) 도메인 기능 계층

기능은 아래 단위로 나뉘어 있습니다.

- auth: 로그인 및 인증 상태
- monitoring: 대시보드 및 현황 조회
- navigation: 메뉴 구조와 역할별 노출 정책
- system-settings: 사용자 운영 관리

이 구조의 장점은 업무 기능이 커져도 기능별로 독립 개발과 유지보수가 가능하다는 점입니다.

#### 4) 상태 및 데이터 계층

- TanStack Query: 서버 데이터 조회, 캐시, 재조회 담당
- Zustand: 메뉴 선택, 필터, 다이얼로그 상태 같은 UI 상태 담당

이 분리는 SaaS 구조에서 중요합니다. 이유는 다음과 같습니다.

- 서버 데이터와 화면 상태가 섞이지 않아 유지보수가 쉬움
- 기능이 늘어나도 상태 오염이 적음
- 캐시 정책을 통일하기 쉬움

#### 5) API 연결 계층

- 모든 API 호출은 apiClient 단일 인스턴스를 사용
- 요청 시 인증 토큰을 자동 주입
- 향후 공통 예외 처리, 로깅, 감사 추적을 붙이기 쉬움

이는 운영 SaaS에서 필수인 표준 연결 계층의 시작점 역할을 합니다.

## 8. 멀티테넌시와 권한 설계

### 8.1 멀티테넌시 구조

현재 구조에는 tenantCode 개념이 이미 포함되어 있습니다.

- 로그인 시 tenantCode 기반 진입
- 사용자 정보에 tenantCode 보관
- 대시보드에서 tenant별 사이트 목록 분기 가능

대표 관점에서 이 의미는 명확합니다.

- 고객사별 서비스 구분이 가능하다
- 사업장별 운영 화면을 분리할 수 있다
- 같은 플랫폼 위에서 여러 조직을 운영할 기반이 있다

### 8.2 권한 구조

현재 권한 구조는 Role Based Access Control 기반입니다.

- 일반 사용자는 기본 운영 화면 중심 사용
- 관리자 계정은 시스템 설정 접근 가능
- 메뉴 노출 자체가 권한 기반으로 제어됨

즉, 단순히 버튼만 숨기는 방식이 아니라 라우팅, 화면 진입, 메뉴 노출이 함께 통제되는 구조입니다.

### 8.3 신규 고객 업체 온보딩 구조

SaaS 플랫폼의 초기 운영에서 가장 중요한 시나리오는 신규 고객 업체가 처음 들어왔을 때의 온보딩 절차입니다.

권장 구조는 아래와 같습니다.

```text
[내부 플랫폼 관리자]
        -> [신규 업체 등록]
        -> [tenantCode 생성]
        -> [업체 관리자 계정 자동 생성]
        -> [최초 접속 수단 발송]
        -> [업체 관리자 첫 로그인]
        -> [업체 사용자 추가 등록]
```

이 구조의 핵심은 사용자 관리보다 먼저 업체 단위 생성이 선행되어야 한다는 점입니다.

- 내부 플랫폼 관리자: 전체 SaaS 운영자
- 업체 관리자: 자기 회사 사용자만 관리하는 고객사 관리자
- 일반 사용자: 실제 현장 점검 및 모니터링 사용자

즉, 관리 주체를 2단계로 분리해야 운영이 깔끔해집니다.

### 8.4 신규 업체 생성 시 시스템이 해야 할 일

내부 플랫폼 관리자가 신규 업체를 등록하면 시스템은 아래 작업을 자동으로 수행해야 합니다.

1. 업체 기본정보 등록
2. 고유 tenantCode 생성 및 중복 검증
3. 기본 사업장 또는 기본 조직 정보 생성
4. 업체 관리자 계정 자동 생성
5. 초기 권한 템플릿 연결
6. 최초 로그인 안내 수단 발송

여기서 가장 중요한 것은 업체 생성과 관리자 계정 발급이 한 번에 연결되어야 한다는 점입니다.

대표 관점에서는 다음 문장으로 설명할 수 있습니다.

"신규 고객사가 생기면 내부 운영자가 업체를 개설하고, 시스템이 그 업체 전용 관리자 계정을 자동 준비해 주며, 이후 고객사 관리자가 자기 회사 사용자를 직접 등록하는 구조입니다."

### 8.5 초기 관리자 계정 발급 원칙

업체 관리자 아이디 자동 생성은 맞는 방향입니다. 다만 초기 비밀번호를 고정값으로 전달하는 방식은 보안상 권장되지 않습니다.

권장 정책은 아래 두 가지 중 하나입니다.

1. 관리자 아이디 자동 생성 + 최초 비밀번호 설정 링크 발송
2. 관리자 아이디 자동 생성 + 임시 비밀번호 발급 후 첫 로그인 시 강제 변경

이 방식의 장점은 다음과 같습니다.

- 보안 사고 위험을 줄일 수 있음
- 고객사 온보딩 절차를 표준화할 수 있음
- 관리자 계정 발급과 최초 접속 통제를 동시에 관리할 수 있음

### 8.6 업체 관리자 첫 로그인 이후 흐름

업체 관리자는 첫 로그인 이후 자기 tenant 범위 내에서만 사용자를 운영해야 합니다.

```text
업체 관리자 로그인
    -> 비밀번호 변경 또는 최초 설정
    -> 기본 정보 확인
    -> 사용자관리 진입
    -> 자기 회사 사용자 조회
    -> 일반 사용자 / 현장 담당자 추가 등록
```

이 단계에서 중요한 설계 원칙은 다음과 같습니다.

- 업체 관리자는 자기 회사 데이터만 조회 가능
- 타 업체 데이터는 조회 자체가 불가해야 함
- 등록한 사용자는 역할에 따라 메뉴가 달라져야 함
- 업체 비활성 상태에서는 로그인 자체를 차단해야 함

### 8.7 현재 구조와 연결되는 이유

현재 프론트 구조에는 이미 이 온보딩을 설명할 수 있는 기반이 있습니다.

- tenantCode 기반 로그인 구조
- roles 기반 접근 제어 구조
- 사용자관리 화면 기반 계정 운영 구조
- 메뉴 권한 제어 구조

즉, 지금은 업체 엔티티와 온보딩 프로세스가 문서상으로 먼저 정의되어야 하는 단계이며, 이후 구현 시에는 내부 운영자용 업체 등록 화면과 첫 로그인 초기화 절차를 추가하면 됩니다.

### 8.8 화면 단위 와이어프레임 확장 방향

온보딩 프로세스를 실제 화면으로 풀면 아래 6개 화면으로 정리할 수 있습니다.

1. 내부 플랫폼 관리자용 신규 업체 등록 화면
2. 업체 생성 완료 및 관리자 계정 발급 확인 화면
3. 업체 관리자 최초 로그인 화면
4. 업체 관리자 초기 설정 화면
5. 업체 관리자용 사용자관리 메인 화면
6. 사용자 추가 등록 다이얼로그 또는 초대 화면

이 화면 구성을 기준으로 실제 온보딩 흐름은 아래처럼 설명할 수 있습니다.

```text
내부 운영자
    -> 업체 등록 화면
    -> 생성 완료 확인 화면

업체 관리자
    -> 최초 로그인 화면
    -> 초기 설정 화면
    -> 사용자관리 메인 화면
    -> 사용자 추가 등록 화면
```

회의에서는 이 6개 화면을 기준으로 "고객사 개설 -> 관리자 계정 준비 -> 첫 접속 -> 사용자 확장" 흐름을 보여주면 가장 직관적입니다.

## 9. 디자인 시스템 관점 구조

현재 화면은 MUI 기반 테마를 사용하며 아래 요소를 공통화하고 있습니다.

- primary, secondary, info, success 기반 색상 체계
- Pretendard 중심 타이포그래피
- 카드, 버튼, 칩, 탭, 셀렉트 등 공통 컴포넌트 변형
- 둥근 모서리와 운영 대시보드형 UI 밀도 유지

대표 회의에서는 이를 다음과 같이 설명하면 됩니다.

"화면을 여러 개 만드는 것이 아니라, 같은 디자인 시스템 위에서 업무 모듈을 계속 얹을 수 있는 구조를 먼저 만들었습니다."

## 10. 현재 구조의 강점

### 10.1 사업 관점 강점

- 핵심 업무 흐름이 로그인, 모니터링, 운영관리로 명확히 구분됨
- 향후 HACCP 문서, 보고서, 분석 기능 확장에 유리함
- 고객사와 사업장 단위 서비스 분리가 가능한 구조임

### 10.2 기술 관점 강점

- feature 단위 구조로 확장성이 높음
- 공통 API 클라이언트로 운영 일관성 확보
- Query와 Zustand 역할 분리로 구조적 안정성 확보
- 권한 기반 메뉴 구조로 운영 보안에 유리함
- 준비중 화면을 통해 단계적 출시 전략 설명이 가능함

## 11. 향후 보완 과제

현재 구조는 방향이 좋지만 SaaS 고도화를 위해 아래 항목이 다음 단계 과제로 적절합니다.

1. 401, 403 응답에 대한 전역 예외 처리 일원화
2. 권한 메타데이터를 백엔드 기반으로 동적화
3. 감사 로그 및 관리자 행위 추적 체계 도입
4. tenant 간 데이터 격리 정책 명확화
5. 권한관리 화면과 승인 정책 화면 구체화
6. 실시간 모니터링과 보고 기능의 실제 데이터 연동 확대

이 항목들은 현재 구조를 뒤엎는 것이 아니라, 이미 마련된 골격 위에 운영 통제 기능을 덧붙이는 성격입니다.

## 12. 회의용 핵심 메시지

대표 회의에서는 아래 메시지 중심으로 설명하는 것이 가장 효율적입니다.

### 메시지 1

현재 프로젝트는 단순 페이지 제작이 아니라 Smart HACCP 운영 SaaS의 기본 플랫폼을 먼저 구축한 상태입니다.

### 메시지 2

로그인, 대시보드, 사용자관리까지의 핵심 운영 흐름은 이미 구조적으로 연결되어 있습니다.

### 메시지 3

메뉴, 권한, 테넌트, 데이터 흐름이 각각 분리되어 있어 고객사별 확장과 기능 확장이 가능한 구조입니다.

### 메시지 4

향후 추가될 CCP 관리, 문서 관리, 분석 리포트 기능은 현재 골격 위에 안전하게 확장할 수 있습니다.

## 14. 결론

현재 SMHC 프론트 구조는 세 가지 목표를 동시에 만족하는 방향으로 설계되어 있습니다.

- 운영자가 바로 사용할 수 있는 화면 구조
- 관리자가 통제할 수 있는 권한 및 설정 구조
- SaaS형 클라우드 서비스로 확장 가능한 기술 구조

즉, 이번 회의에서는 화면의 완성도만 보는 것이 아니라, 이 프로젝트가 향후 여러 고객사와 여러 공정을 수용할 수 있는 서비스 골격을 갖췄다는 점을 중심으로 설명하는 것이 적절합니다.
